⎿Ollama API — /api/chat With think: true, reasoning models emit chunks shaped: 
{ "model": "gpt-oss:20b",
  "message": { "role": "assistant", "thinking": "...", "content": "" },
  "done": false }
On the final chunk the message.content field carries the answer when the model uses the content channel, OR remains empty if the model used the thinking-only channel (Harmony reasoning-only).

For models that route the final output through the thinking channel (gpt-oss:20b often does this for short answers), callers must accept message.thinking as the actual response text when message.content is never populated.

Sources confirm the diagnosis. Two compounding bugs in our adapter:

  1. OllamaProvider.ts:696 — when thinkingText equals contentText we silently drop the text_delta. That dedup made sense when both channels were emitted; it's a leak when gpt-oss only emits via thinking AND happens to mirror.
  2. OllamaProvider.ts:733 (terminal chunk) — when accumulatedContent === '' AND accumulatedThinking is non-empty, we yield zero text_deltas. The Anthropic Messages contract requires at least one text_delta for the assistant turn to be non-empty (docs.claude.com/streaming#event-types).

Right layer to fix: OllamaProvider (the producer), not the route counter (the consumer). I'll dispatch a sub-agent to capture a real gpt-oss NDJSON fixture, then write the red test, then fix.
