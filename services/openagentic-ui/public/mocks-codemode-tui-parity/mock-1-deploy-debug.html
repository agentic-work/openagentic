Title: code-manager: clean up orphan-named Secrets on session-start after hashUserId rename

Body: 
## Problem
After renaming `hashUserId` from 16→12 char slice
(commit c4e2f871), Kubernetes Secrets created
under the old shape (ws-<16hex>-creds) are
orphaned in tenant namespaces. They don't break
anything but they bloat the namespace and confuse
operators reading `kubectl get secrets`.

## Repro
Any tenant that ran a codemode session under
cm 0.6.27-or-prior + then upgraded to 0.6.28
will have one orphan per active user.

## Fix
In `openagentic-manager/src/services/userStorage.service.ts`
`startSession()`, after creating the new Secret,
list all Secrets matching `ws-*-creds` for the
tenant namespace, drop any whose name doesn't
match the canonical 12-char shape derived from
`hashUserId(userId)`. Log the cleanup at INFO.

## Acceptance
- Test: spawn session with stale 16-char Secret
  pre-existing → after `startSession` only the
  12-char Secret remains.
- No-op when no orphans present.

## Severity
P3 — cosmetic. Not a security bug; orphan
Secrets contain only this user's bucket creds.
