SOC 2 Type II is an attestation of operating effectiveness over a period, not a point-in-time design review. The deliverables the user needs are: (a) a control inventory, (b) per-control evidence artifacts for a ≥6-month window, (c) mapped gaps and concrete remediation tickets with owners and due dates, and (d) a repeatable collection playbook so the next audit isn't a fire drill.

The 2017 TSC framework has 64 common criteria across 5 categories (CC Security is always in scope; Availability, Confidentiality, Processing Integrity, Privacy are opt-in). Based on this stack — SaaS, multi-cloud, customer data — I'll scope Security + Availability + Confidentiality. That's typically 48-54 controls in the final audit matrix.

Plan:

  • Pass 1 — inventory each cloud in parallel. aws-inventory, azure-inventory, gcp-inventory run concurrently. Each pulls IAM, logging, encryption posture, backup state, and change-management evidence.
  • Pass 2 — control-mapper takes the merged inventory and maps each finding to the TSC 2017 control it proves or breaks. This is where we find gaps.
  • Pass 3 — evidence-collector writes the playbook (what to export + at what cadence). ticket-writer emits a ticket per gap with owner, severity, estimated effort. policy-gap-analyzer double-checks against our existing policy library.

I'll budget ≤$0.70 and <80s. If control-mapper raises a framework-drift question (SOC 2 2017 vs. SSAE 18 mapping) I'll make that a footnote rather than a sub-agent pass.

Executive summary

You are 57% of the way to a clean SOC 2 Type II audit across the scoped trust-service categories. Out of 54 in-scope controls, 31 pass today, 15 have operating-effectiveness gaps (control exists but wasn't applied consistently), and 8 are design gaps (control doesn't exist yet). The design gaps are the long poles; operating-effectiveness issues can all be closed with configuration changes in the next 4 weeks.

The auditor will sample. For Type II, they pick a window (usually 6 months) and pull evidence from random points in that window to prove each control operated continuously. That's why the evidence-collection playbook is non-negotiable — missing a weekly IAM credential report for one week in the window means the auditor cannot form an opinion on CC6.2. The playbook I wrote codifies exactly which artifacts to produce, at what cadence, and where they live. If you follow it for 90 days you'll have a clean Type II sample envelope.

Recommended close order

I would close gaps in three waves tied to the 3 audit-prep sprints we have before a July 2026 Type II window:

  • Sprint 1 · weeks 1-2: all critical gaps (SOC2-101 → SOC2-103) plus re-enable marketing-account CloudTrail. These are all ≤5 pts and block any clean opinion letter. No policy or training work — pure config changes.
  • Sprint 2 · weeks 3-6: all high-severity operating gaps (KMS rotation, TLS 1.2 minimum, GuardDuty in missing regions, GCP audit-log sink). Requires cross-team coordination but no architectural work. Begin the evidence-collection cadence in parallel so the 6-month window starts producing samples.
  • Sprint 3 · weeks 7-12: the design gaps. Backup Vault SCP-enforced retention, conditional-access enforcement (move from report-only to enforce), BigQuery CMEK migration, change-management tooling for the 4% of PRs missing approver records. Higher-risk changes because they affect developer velocity; stage each behind a feature flag.
