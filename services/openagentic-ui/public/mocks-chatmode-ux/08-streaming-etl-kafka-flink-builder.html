Streaming ETL has four invariants I want to hold from the first line of code:

  • Exactly-once end-to-end. That's Kafka transactional reads + Flink checkpoints + ClickHouse idempotent inserts (via dedup key). Cheap to design for, expensive to retrofit.
  • Schema evolution without replaying history. Avro + Schema Registry with BACKWARD compat. Fields only get added, never renamed.
  • Backpressure that doesn't melt the cluster. Flink's default is “pause upstream”, which can stall Kafka consumer groups. I'll configure akka.ask.timeout + taskmanager.memory.network explicitly so the dataplane degrades gracefully.
  • Late-data handling. Windowed aggregates need a watermark strategy; 10-min bounded-out-of-orderness seems right for most event types, with a side-output for beyond-watermark events so we don't drop data silently.

Four sub-agents: codegen (the Java/Scala Flink job + TF modules), security (secrets + mTLS), testing (integration test with testcontainers), docgen (runbook + schema docs).

Operational shape

  • Checkpointing: 10-second interval, RocksDB state backend, MinIO-compatible object store. Restart-from-checkpoint tested (FailureRecoveryIT).
  • Watermark: BoundedOutOfOrdernessWatermarkStrategy(Duration.ofMinutes(10)). Late events route to orders_late side-output and a separate ClickHouse table.
  • Backpressure: TaskManager network memory raised to 512MB; akka.ask.timeout 30s; if ClickHouse stalls >60s the job backs off Kafka consumption rather than OOMing.
  • Schema evolution: Confluent Schema Registry with BACKWARD_TRANSITIVE. Breaking changes (field rename/remove) are rejected at the registry; additive changes flow through without job restart.

What I'd iterate on next

  • Add Prometheus kafka_lag_seconds alert that fires at 60s sustained; runbook kafka-lag.md gives SRE two diagnosis commands + rollback.
  • Enable ClickHouse async insert mode once we confirm Clouds support it; should drop p99 insert latency another 30%.
  • The Terraform module pins MSK to 3 brokers. If traffic exceeds ~40 MB/s per topic we should scale to 6 and re-run the partition-rebalance migration — procedure documented in runbooks/msk-scale.md.
