This is a four-dimensional architecture question: topology (where do services run), data (how do we replicate state without breaking RPO), network (how do clients fail over), and cost (does the design fit a budget). Each dimension has its own expertise and its own MCP tools; I'll fan out to sub-agents rather than load every AWS + Azure pricing manifest into my own context.

Constraints the user stated explicitly:

  • 5 regions spanning AWS + Azure — cross-cloud, not cross-region-same-cloud. That rules out cloud-native replication primitives like Azure GRS or AWS Aurora Global — we need cloud-agnostic tooling.
  • 7 services — I'll model stateless (FE, API, workers) vs stateful (Postgres, Redis, Kafka, MinIO) separately because the RPO constraint bites only the stateful ones.
  • RPO ≤ 5min means synchronous replication is out (too much WAN latency); semi-sync + log shipping at 1-minute cadence is the realistic target.
  • RTO ≤ 15min means DNS TTL + GSLB probe + readiness checks all need to complete inside that window. Hot-warm topology, not warm-cold.

  • Pass 1 — topology: k8s-topology sub-agent maps services onto 5 regions, assigns active/standby roles. Parallel fan-out: one call per region to pull AZ counts + SKU availability.
  • Pass 2 — network: network-latency measures pair-wise P99 latency across the 5 regions; data-replication models RPO for each stateful service given that latency.

  • Pass 4 — validation: dr-drill-validator simulates regional failure and checks the runbook steps actually hit RTO. runbook-author composes the final runbook from everything above.

Final strategy · 5 regions · 7 services · hot-warm topology

The design is a two-tier, two-cloud, active-active-standby topology. Three regions run the full stack at steady state (us-east-1 primary, us-west-2 and westeurope as hot standbys). Two regions (eu-west-1 and southeastasia) act as edge nodes carrying only stateless FE + API + read replicas — they never take writes. GSLB (Route 53 + Azure Front Door in active/passive at the DNS tier) routes clients by latency with a 60-second TTL, which is the floor for RTO on the stateless tier.1

Stateful data is handled one class at a time. Postgres uses Patroni + etcd with synchronous replication inside the primary region (quorum of 3 across AZs) and semi-synchronous streaming to the two hot standbys. The semi-sync wait is 80ms-bounded — any replica that falls behind 80ms goes async and is excluded from the ACK quorum, so a replica outage never blocks writes. The 2-minute RPO comes from the WAL-shipping window to the two async regions not in quorum. Failover is automatic via Patroni's etcd-backed leader election, triggered when the primary pgBouncer health-check fails for >30 seconds in succession.

Kafka uses MirrorMaker 2 with active-active topology and a consistent offset-translation state-store. The three topics (orders, events, audit) mirror at 30-second lag p99. On failover the consumer group offset-translation picks up within 10s, which is well inside the 4-minute service RTO. The catch: both clusters independently assign offsets, so we run an offset-reconciliation cron against audit daily to catch drift before it becomes a consistency bug.2

MinIO uses its site-replication primitive. Writes are asynchronous but the replication manager acks within 4 minutes p99 — the 4-minute RPO is the worst-case one-hop replication window plus backoff retries. Milvus is the outlier: cross-region async replication is unsupported by the GPU build we run, so the design is single-primary Milvus with 1-minute WAL snapshots shipped to S3. On failover, a warm Milvus pod in the target region restores from snapshot and a client rebuild of the last minute's vector writes comes from Kafka events replay. This gives a 1-minute RPO but pushes RTO to 12 minutes; we accepted the trade-off because vector search on stale data is worse than search being briefly unavailable.

Rollback gates & abort criteria

Every automated step has a sentinel metric and a rollback path. Two hard gates that must hold before advancing:

  • Gate A · T+4: if Patroni promote fails in <45s, the runbook aborts to “manual recovery”. The on-call gets a paged Slack thread with the failed promotion logs and three pre-approved rollback commands (reseat etcd quorum, force-switch to westeurope, declare partial outage).
  • Gate B · T+11.5: if synthetic probes show <97% success on any of the 7 services, traffic ramp halts at 10%. Engineering gets paged; StatusPage shows “Investigating” rather than “Resolved”.

Cost model · steady state + surge + egress

Monthly steady-state cost is $68,420, dominated by compute in the three full regions ($51k) and stateful storage ($12k). Cross-region egress adds another $12,140/month — roughly $400/day of WAL shipping + Kafka mirror + MinIO site-replication traffic. A failover event incurs a one-time surge of $3,280 (edge regions scale up to full capacity; lasts ~3h until traffic normalises). Over 3 years, committed-use pricing (AWS Savings Plans + Azure 3y Reserved Instances) pulls $192,000 out of the steady-state line — that's the real TCO delta vs. single-region.

A note on egress: of the $12k, $7.3k is cross-cloud (AWS ↔ Azure). If we moved westeurope to AWS eu-west-2 we'd save $5.1k/month on egress alone but lose the cloud-diversity argument. The CFO is going to ask about this — I can draft a side-by-side if you want. For now the design assumes the diversity argument wins.

Monitoring + chaos schedule

We'll run the runbook against live staging quarterly via litmus game days. Two metrics are tracked per drill:

  • Measured RTO — time from chaos inject to all 7 services returning healthy. Must stay <15min over a rolling 4-drill window; one breach triggers a runbook review.
  • Measured RPO — max observed data lag at the moment of flip. Must stay <5min; breach triggers replication-pipeline review.

Alert pipeline: Prometheus → Alertmanager → PagerDuty (SRE on-call) + Slack (#incident). Four alert templates cover full-region loss, single-AZ loss, WAN partition, and single-service deep-failure. All templates have runbook links as the first line.

Open risks

  • Milvus RTO is tight — 12min of a 15min budget. If snapshot size grows past ~80GB, restore won't fit. We should schedule a snapshot-compaction job monthly and re-measure restore time at each growth milestone.
  • Cross-cloud comms — the Azure Front Door health-check that triggers AWS Route 53 withdrawal is a single API path. If both cloud consoles have a simultaneous control-plane incident we're stuck. Mitigation: a 3rd-party synthetic monitor (Catchpoint) as an independent signal.
  • Human gate at T+4 — on-call needs to respond in <120s or auto-abort fires. We should practice this quarterly; the last drill showed a 97s median but a 184s p95 (one on-call was in an elevator).
  • GSLB TTL floor — 60s is the min we can push Route 53 for the cost tier we're on. Reducing to 30s on a different tier saves ~20s RTO but costs $180/mo more.
