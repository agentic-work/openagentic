The user wants a full right-sizing audit. I need to plan this carefully because Azure VM metadata lives in ARM and CPU time-series lives in Azure Monitor — two different APIs, and I only know one subscription context per call.

Step plan:

  • First, enumerate subscriptions the user's identity can reach — azure_list_subscriptions. If this returns >1 sub, I'll fan out the next call per-sub in parallel.
  • For each sub, pull the VM inventory via Resource Graph (one KQL query is cheaper than list-by-rg). Grab size, location, power state, OS. I also want VNet context in case we recommend moving SKUs between regions.
  • For each running VM, pull Percentage CPU over 30d at 1h granularity from Azure Monitor. P95 and average are what matter for right-sizing — avg alone underweights bursty workloads.
  • Delegate the right-sizing + pricing math to a cost-analysis sub-agent. It has access to the Azure Retail Pricing API and the SKU family compatibility matrix I don't want to load into my own context.
  • Compose the final answer: top-line savings, per-VM recommendations, caveats (reserved instances, savings plans).

Cost ceiling for this task should be under $0.10. Budget: ~3k output tokens plus ~1k sub-agent tokens.

Summary

8 of 17 VMs are provably oversized against their 30-day P95 CPU utilisation. Right-sizing these saves $2,847/month (46% reduction on the affected subset, 23% on the full fleet). None of the recommended moves require downtime — all target SKUs are in the same family & region1.

Recommended actions

  • Resize 6 VMs in prod (vm-api-blue-01/02, vm-redis-cache-01, vm-grafana-01, and 2 more) down one tier during a scheduled maintenance window. All moves are in-family so no OS disk rebuild.
  • Stop vm-jumpbox-prod outside business hours (it averages 0.4% CPU). Use a scheduled start/stop automation — Bastion covers most operator needs.
  • Retire vm-dev-sandbox-01 or migrate the tenant to the shared dev cluster. Standalone D8 for a 1% CPU workload is indefensible.

Caveats

  • P95 was hit during a single weekly ETL window on vm-worker-etl-01 — I left it at E16s; if that window tolerates slower completion we could drop a tier and save another ~$180/mo.
  • These prices assume Pay-As-You-Go. If you have a 3-year RI on D8s_v5, the cheapest move may be to sell rather than resize2.
  • No CPU-credit-bank SKUs (B-series) were recommended for prod workloads — they're brittle under sustained load even if avg is low.
