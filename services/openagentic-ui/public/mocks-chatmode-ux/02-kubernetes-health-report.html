3 · Detailed findings

3.1 · Milvus memory pressure

The milvus-standalone-0 pod in agentic-prod has OOM-killed 12 times in 7 days, concentrated in the 09:00–11:00 UTC window (coincides with nightly index compaction + morning query spike).

Memory request: 8Gi · Memory high-watermark observed: 11.2Gi · Peak working set before OOM: 12.8Gi.

Recommended action: raise the request/limit to 16Gi (current node has 32Gi so no repack needed) and enable the rocksmq WAL size cap in values-k3s-local.yaml. This is a low-risk change; rolling restart during the 02:00 UTC maintenance window is appropriate.

3.2 · API CPU throttling

Over the 30-day window, rate(container_cpu_cfs_throttled_periods_total[5m]) was non-zero on the openagentic-api ReplicaSet for 18% of 5-minute buckets. The workload nodepool (Standard_D8s_v5) averages 72% CPU at p95; the next autoscaler tick would bring up another node but hasn't triggered because aggregate utilisation is under the scale-up threshold.

Options in increasing cost: (a) raise individual pod CPU request from 500m to 1000m (most efficient), (b) lower the HPA scale-out threshold from 75% to 65%, (c) add a third node. We recommend (a) first.
