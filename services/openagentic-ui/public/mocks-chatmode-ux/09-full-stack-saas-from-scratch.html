This is a full-stack scaffold: front-end, back-end, data layer, billing, auth, infra, CI. The user wants it all to work together on day one, which means I can't just emit 15 separate files and hope. I'll plan around four coupling points that always cause pain if rushed:

  • Tenant isolation is the root invariant. Every SQL query, every cache key, every Stripe customer reference, every S3 prefix has to respect tenant_id. I'll use Postgres RLS as the ground-truth enforcement and add Prisma middleware on top so the Node layer sets SET app.current_tenant per request. RLS bypass is the #1 security bug in multi-tenant SaaS; I want the database to be the one saying no.
  • OIDC + JWT + session lifecycle. OIDC handshake returns ID token, I issue an app JWT with tenant + role claims. Refresh via rotating refresh-token table. No in-memory sessions — stateless Fastify so we can scale horizontally.
  • Stripe lifecycle. Subscription lifecycle is the hard part. Webhook → reconcile into Postgres → feature flags. Idempotent handler using event.id as the dedup key. Separate webhook-receiver route — must NOT require auth but must validate Stripe signature.
  • CI/CD and k8s. Github Actions → build images → push to GHCR → ArgoCD pulls → k8s rolling deploy with HPA. Helm chart with reasonable defaults.

4 sub-agents in parallel where possible: codegen (Next.js-ish monorepo), security (RLS + JWT + webhook validation), testing (vitest + Playwright E2E for checkout), docgen (README + API docs + runbook).

Expect pass 2 to fix: likely RLS policy gap (the common bug is forgetting to include RLS on a derived table), maybe a webhook retry-loop bug, probably a Prisma-middleware order issue where logging runs before tenant is set.

What's in the repo

  • apps/web — React + Vite + Tailwind. Routes: pricing, sign-up, dashboard, billing. Uses TanStack Query to fetch with the app-JWT in Authorization.
  • apps/api — Fastify 4. Route groups: /auth (OIDC callback + JWT mint), /billing (Stripe checkout + webhook), /api/v1 (business domain). Tenant-scope plugin runs before any business route and refuses requests without a valid tenant in the JWT.
  • packages/db — Prisma schema + RLS migrations. 14 business tables, each with tenant_id UUID + RLS policy USING (tenant_id::text = current_setting('app.current_tenant', true)).
  • packages/ui — shared React components (buttons, forms, pricing card) built with tsup so the web app imports compiled JS in prod.
  • helm/ — umbrella chart with sub-charts for web + api. HPA on CPU (target 70%), PDB (minAvailable 1), NetworkPolicy (api egress only to postgres + redis + stripe + oidc provider).
  • .github/workflows/ — PR: build + lint + typecheck + test + e2e (with testcontainers Postgres + Redis). Main: image build + push to GHCR + ArgoCD bump.

Security posture

  • RLS everywhere — Postgres-level enforcement, not app-level. Any future table MUST include the policy or CI fails (migration linter checks it).
  • JWTs are app-level, not OIDC id-tokens. ID token used once to establish session then discarded; app-JWT contains tenantId, role, userId. 1h TTL, refresh via rotating table.
  • Webhook integrity — Stripe signature validated before body parse; event.id idempotency via ProcessedWebhook table; 400 on bad sig so Stripe retries are surfaced as failures rather than silently swallowed.
  • Rate limiting — tenant-scoped via Redis, not IP-scoped. IP-scoped limits on a multi-tenant SaaS punish the wrong people.
  • Secrets — all via env; .env.example documents each one; GitHub Actions uses secrets.*; k8s manifests reference ExternalSecret CRDs (AWS SM / Vault plug-compatible).

What I didn't do

  • Production-grade observability. OTel wiring is there but no dashboards. I can generate a Grafana JSON next if you want.
  • Email + verification. Sign-up via OIDC assumes the identity provider's email is trusted. If you need a custom sign-up flow with your own verification email, that's a separate task.
  • Admin console. Only tenant-level roles (owner, admin, member). A cross-tenant admin view for support would be a new app under apps/admin with a separate RLS-bypass role.
  • Seed data. The DB boots empty. I can add a pnpm seed with demo tenants + a few fake subscriptions if you want something to click around.
