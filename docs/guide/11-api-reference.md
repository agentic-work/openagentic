# API Reference

This page documents the **OpenAgentic Platform API** — the REST surface exposed by the `openagentic-api` service. Everything the UI does (login, chat, Flows, admin, RAG) goes through this API, and you can drive it directly with `curl`, the OpenAI SDK, or any HTTP client.

It covers the authentication model, the core endpoints (health, chat, admin, workflows, system config), the streaming wire format, the OpenAI-compatible shim, and a complete end-to-end example.

> **Grounded in source.** Every endpoint, field, and default below is read from `services/openagentic-api/src/routes/` and the Fastify plugins in `services/openagentic-api/src/plugins/`. If something is not listed here, treat it as not part of the supported surface.

---

## Base URL and conventions

| | |
|---|---|
| **Default base URL** | `http://localhost:8080` (the UI/API host port, `UI_HOST_PORT`) |
| **Content type** | `application/json` for request and response bodies, except streaming chat (see [Streaming wire format](#streaming-wire-format)) |
| **Path prefix** | All platform routes are under `/api/*`. The OpenAI-compatible shim lives under `/api/v1/*`. |
| **Interactive docs** | Swagger UI is served at **`/api/swagger`**; the raw OpenAPI 3.1 spec is at `/api/swagger/json`. The admin-gated spec is also at `/api/openapi.json`. |

Errors use a consistent envelope:

```json
{ "error": "Invalid credentials" }
```

Some endpoints return a richer shape (`{ "success": false, "error": "...", "message": "..." }`); admin endpoints commonly return `{ "success": true, ... }`. HTTP status codes are authoritative (`401` unauthenticated, `403` forbidden / admin-required, `404` not found, `429` rate-limited, `5xx` server error).

## Authentication

The OSS edition is **local-auth only** — a single local user store (username/password) backed by JWTs and optional API keys. There is no SSO / Entra ID / OAuth in the open-source edition.

There are two ways to authenticate a request, handled by the unified auth middleware (`src/middleware/unifiedAuth.ts` → `src/auth/tokenValidator.ts`):

| Method | Header | Used for |
|---|---|---|
| **Bearer JWT** | `Authorization: Bearer <jwt>` | Interactive / session-based access (what the UI uses) |
| **API key** | `X-API-Key: <key>` | Programmatic / headless access |

> The API key may also be passed as `Authorization: Bearer oa_...` — keys carry the `oa_` prefix, so the validator routes them to key validation regardless of which header you use.

### Getting a JWT — `POST /api/auth/local/login`

This is the local login endpoint (registered at the `/api/auth/local` prefix). No auth required.

**Request**

```json
{
  "username": "admin@openagentic.local",
  "password": "<password-from-install-wizard>"
}
```

`username` accepts the user's email. The password for the seeded admin is generated by `install.sh` / the setup wizard.

**Response `200`**

```json
{
  "success": true,
  "token": "<jwt>",
  "user": {
    "id": "…",
    "email": "admin@openagentic.local",
    "name": "…",
    "isAdmin": true,
    "groups": []
  }
}
```

The returned `token` is a JWT signed with the platform `JWT_SECRET` / `SIGNING_SECRET`, with a payload of `{ userId, email, name, isAdmin }` and a **24-hour** expiry. A server-side session row is also created.

**Other responses**

| Status | Meaning |
|---|---|
| `400` | `username` and `password` are both required |
| `401` | `{ "error": "Invalid credentials" }` — unknown user or bad password |
| `403` | `{ "error": "Password change required", "requiresPasswordChange": true, ... }` — the seeded admin must change its password on first login when `ADMIN_REQUIRE_PASSWORD_RESET` is not `false` |

### Other local-auth endpoints

These are all under the `/api/auth/local` prefix (`src/routes/local-auth.ts`):

| Method | Path | Auth | Purpose |
|---|---|---|---|
| `POST` | `/api/auth/local/login` | none | Log in, get a JWT |
| `POST` | `/api/auth/local/logout` | Bearer | Invalidate active sessions for the user |
| `GET` | `/api/auth/local/me` | Bearer | Return the current user profile (`{ user: { id, email, name, isAdmin, groups, createdAt } }`) |

### Inter-service / token verification

`src/routes/auth.ts` exposes verification endpoints used for inter-service token checks and the legacy profile path:

| Method | Path | Purpose |
|---|---|---|
| `POST` / `GET` | `/api/auth/verify` | Verify a token |
| `POST` | `/api/auth/validate-token` | Validate an inter-service token |
| `GET` | `/api/auth/me` | Current-user profile (legacy) |
| `GET` / `POST` | `/api/auth/logout` | Logout |

### API keys

API keys are created and managed by an admin and used for headless access. Keys have the format `oa_<43-char base64url>` (user keys) and `oa_sys_<...>` (system / inter-service tokens). They are stored as bcrypt hashes — the plaintext key is shown **once** at creation time. Management endpoints (`src/routes/admin-api-tokens.ts`, admin-only):

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/admin/tokens` | List API tokens (`?userId=`, `?includeExpired=true|false`) |
| `POST` | `/api/admin/tokens` | Create a token. Body: `{ userId, name, expiresInDays?, rateLimitTier?, rateLimitPerMinute?, rateLimitPerHour?, rateLimitBurst? }` |
| `DELETE` | `/api/admin/tokens/:id` | Revoke a token |
| `GET` | `/api/admin/tokens/users/available` | Users eligible for token issuance |
| `GET` | `/api/admin/tokens/metrics` | Token usage metrics |

Use the returned key in `X-API-Key` on subsequent calls.

### Rate limiting

Requests are rate-limited per authenticated user (falling back to client IP). Limits are admin-configurable live via the admin console (stored in Redis at `platform:rate_limits`). When Redis is unavailable the fallback is **600 req/min for admins, 120 req/min for everyone else**, over a 1-minute window. Internal-service calls, `/health` / `/api/health`, and WebSocket upgrades are exempt. Exceeding the limit returns `429` with `{ "statusCode": 429, "error": "Too Many Requests", "message": "..." }`.

## Smart Model Router — do not send a `model` field

The platform runs a **SmartModelRouter** that selects the model for every chat turn based on prompt complexity, tool usage, and required capabilities (e.g. vision). It routes *down* to a cheap local model for trivial prompts and *up* for hard ones, all against the registry-driven model list (no model IDs are hardcoded in the platform).

**Do not pass a `model` field in chat request bodies.** Although the `/api/chat/stream` schema technically accepts an optional `model`, resolution precedence (`src/routes/chat/resolveChatModel.ts`) is:

1. Explicit `model` in the request (discouraged — bypasses routing)
2. The model persisted on the session row
3. **SmartModelRouter's pick** (the intended path)
4. The DB-backed default chat model
5. An emergency `default` sentinel

Leave `model` out and let the router do its job. This is the supported and recommended behavior.

## Health

Health endpoints (`src/routes/health.ts`, plus `src/plugins/health.plugin.ts`). The basic `/api/health` requires no auth — it is the canonical liveness/readiness probe.

| Method | Path | Auth | Purpose |
|---|---|---|---|
| `GET` | `/api/health` | none | Basic health + dependency status |
| `GET` | `/api/health/detailed` | none | Health plus DB query self-tests |
| `GET` | `/api/health/comprehensive` | none | Full check: DB, chat model, embedding model, MCP orchestrator, Milvus vector store |
| `GET` | `/metrics`, `/api/metrics` | none | Prometheus metrics (text exposition format) |
| `GET` | `/model-health` | none | Live chat-model health probe |

**`GET /api/health`** returns `200` when the database is reachable, `503` otherwise:

```json
{
  "status": "healthy",
  "timestamp": "2026-06-18T12:00:00.000Z",
  "version": "1.0.0",
  "commit": "dev",
  "build": "…",
  "database": { "status": "connected", "method": "prisma" },
  "redis": { "status": "connected" },
  "milvus": { "status": "connected" },
  "users": { "count": 1 }
}
```

`/api/health/comprehensive` returns `200` only when every subsystem is healthy, `503` if any check fails, with a per-check breakdown under `checks.{database,chat_model,embedding_model,mcp_orchestrator,vector_storage}`.

## Chat

The chat API (`src/routes/chat/index.ts`, mounted at `/api/chat`) is the core conversational surface. All endpoints except the debug/health helpers require authentication (Bearer JWT or API key).

### Session management

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/chat/sessions` | Create a chat session. Body: `{ title?, model?, metadata? }`. Returns `201` with the session object. |
| `GET` | `/api/chat/sessions` | List sessions. Query: `limit`, `offset`, `sortBy` (`updated_at`\|`created_at`), `sortOrder` (`asc`\|`desc`). Returns `{ sessions: [...], total }`. |
| `GET` | `/api/chat/sessions/:sessionId` | Get one session |
| `PUT` | `/api/chat/sessions/:sessionId` | Update a session (title, etc.) |
| `DELETE` | `/api/chat/sessions/:sessionId` | Delete a session |
| `GET` | `/api/chat/sessions/search` | Search sessions |
| `GET` | `/api/chat/sessions/:sessionId/messages` | List messages in a session |
| `GET` | `/api/chat/sessions/:sessionId/messages/:messageId` | Get one message |
| `DELETE` | `/api/chat/sessions/:sessionId/messages/:messageId` | Delete one message |

> **Note:** `POST /api/chat/sessions/:sessionId/messages` does **not** send a message — it returns a `301` pointing you at `POST /api/chat/stream`. Always send messages via the streaming endpoint.

### `POST /api/chat/stream` — send a message, stream the reply

This is the main chat endpoint. It requires authentication and **streams** the response as NDJSON (see [Streaming wire format](#streaming-wire-format) below).

**Request body**

```json
{
  "message": "Which pods are crash-looping in the prod namespace?",
  "sessionId": "session_…",
  "groundingEnabled": false
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `message` | string | yes | User message content |
| `sessionId` | string | yes | Target session ID (create one first via `POST /api/chat/sessions`) |
| `model` | string | no | **Leave unset** — the Smart Router selects the model |
| `attachments` | array | no | `{ originalName, mimeType, size, data }` with base64 `data` |
| `files` | array | no | `{ name, type, content }` |
| `promptTechniques` | string[] | no | Optional prompt-engineering technique hints |
| `groundingEnabled` | boolean | no | Per-turn web-grounding opt-in (defaults `false`) |

The model autonomously discovers and invokes MCP tools mid-turn (the V2 discovery pipeline), so tool calls against AWS, Kubernetes, Prometheus, Loki, GitHub, etc. happen inside this single streamed turn. Mutating tool calls pause behind the [approval gate](#human-in-the-loop-approval).

### Discovery and capability helpers

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/chat/tools` | MCP tools available to the user |
| `GET` | `/api/chat/mcp/status` | MCP orchestrator status |
| `GET` | `/api/chat/mcp-functions` | MCP tools projected as function definitions |
| `GET` | `/api/chat/models/available` | Models the user may select |
| `GET` | `/api/chat/models/default` | Default model |
| `GET` | `/api/chat/capabilities` | Model/feature capabilities |
| `POST` | `/api/chat/knowledge/ingest` | Ingest a document into the RAG knowledge base |
| `POST` | `/api/chat/knowledge/search` | Semantic search over the knowledge base |
| `POST` | `/api/chat/generate-image` | Image generation via the `imageGen` role |

### Human-in-the-loop approval

Mutating tool calls pause and emit an approval event on the stream; the client resolves them out-of-band.

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/chat/tool-approval/:requestId` | Approve/deny a pending tool call (chat-scoped) |
| `GET` | `/api/chat/tool-approvals/pending` | List pending approvals |
| `POST` | `/api/approvals/:auditId/approve` | Approve a gated tool call by audit-row ID |
| `POST` | `/api/approvals/:auditId/deny` | Deny a gated tool call by audit-row ID |

The approve/deny endpoints write the immutable audit-row transition and resolve the in-process approval registry so the paused chat turn resumes. Response: `{ ok: true, auditId, approved, decided, resolved }`.

## Streaming wire format

`POST /api/chat/stream` (and other platform streams) use **NDJSON**, not SSE.

- **Content-Type:** `application/x-ndjson`
- One JSON object per line, terminated by `\n` (no `event:` / `data:` framing, no `\n\n` delimiter).
- Each object carries a `type` field. Clients split on `\n`, `JSON.parse` each complete line, and switch on `type`.
- Proxy buffering is disabled at the response (`X-Accel-Buffering: no`, `Cache-Control: no-cache`, `Transfer-Encoding: chunked`). Keepalive `ping` frames are emitted every 3 seconds.

Reference: `src/infra/ndjson.ts`.

**Example frames**

```
{"type":"stream_start","sessionId":"session_…"}
{"type":"text_delta","content":"Checking the prod"}
{"type":"text_delta","content":" namespace…"}
{"type":"thinking_complete", ...}
{"type":"stream_complete", ...}
```

Common frame types include `stream_start`, `text_delta` / `content_delta`, `thinking_complete`, `completion_complete`, `stream_complete`, `ping`, and `error`. Tool-call and sub-agent activity is interleaved as additional typed frames. An `error` frame uses `{ "type": "error", "code": "...", "message": "...", "timestamp": "..." }`.

> The OpenAI-compatible endpoint (`/api/v1/chat/completions`) intentionally stays on **SSE**, because that is the public API contract clients expect. The NDJSON format applies to the internal `/api/chat/*` streams.

```json

  "workflowEngine": { "type": "native", "name": "OpenAgentic Workflows", "available": true },
```

## End-to-end example

A complete flow: log in, create a session, stream a chat turn. Note that **no `model` field is sent** — the Smart Router picks the model.

```bash
BASE=http://localhost:8080

# 1) Log in as the seeded admin → capture the JWT.
TOKEN=$(curl -sX POST "$BASE/api/auth/local/login" \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin@openagentic.local","password":"<from-wizard>"}' \
  | jq -r .token)

# 2) Create a chat session → capture its ID.
SESSION=$(curl -sX POST "$BASE/api/chat/sessions" \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"title":"Ops check"}' \
  | jq -r '.id // .session.id')

# 3) Send a message and read the NDJSON stream.
#    (No "model" field — the Smart Router chooses.)
curl -N -sX POST "$BASE/api/chat/stream" \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d "{\"sessionId\":\"$SESSION\",\"message\":\"Which pods are crash-looping in prod?\"}"
```

The stream emits one JSON object per line. A minimal consumer:

```bash
curl -N -sX POST "$BASE/api/chat/stream" \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d "{\"sessionId\":\"$SESSION\",\"message\":\"Hello\"}" \
| while IFS= read -r line; do
    echo "$line" | jq -r 'select(.type=="text_delta") | .content' 2>/dev/null
  done
```

### Using an API key instead

For headless access, an admin creates a key, then you use it with `X-API-Key`:

```bash
# Admin creates a key for a user (returns the plaintext key once).
curl -sX POST "$BASE/api/admin/tokens" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"userId":"<user-id>","name":"ci-bot","expiresInDays":90}'

# Then call the API with the key.
curl -sX POST "$BASE/api/chat/stream" \
  -H "X-API-Key: oa_…" \
  -H 'Content-Type: application/json' \
  -d "{\"sessionId\":\"$SESSION\",\"message\":\"status report\"}"
```
