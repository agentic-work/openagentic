# Configuration

OpenAgentic is configured entirely through environment variables (Docker Compose) or
structured Helm values (Kubernetes). There are **no configuration files to hand-edit
inside the containers** — everything environment-specific flows through `.env` (compose)
or `values.yaml` (Helm).

This reference is grounded in the shipped `.env.example`, `docker-compose.yml`, and
`helm/openagentic/values.yaml`. Every variable, default, and behavior below is exactly
what the code reads.

Two ground rules apply across the whole platform:

- **No weak secret defaults ship.** Compose requires the trust-root secrets via
  `${VAR:?...}` (a bare `docker compose up` fails fast if any is unset); the Helm chart
  `required`-guards each one (an unset value aborts `helm install`). `install.sh`
  generates strong random values with `openssl rand -hex 32` on first run.
- **No hardcoded model IDs.** The active chat/embedding/image models are registry- and
  seeder-driven, never baked into source. You point the platform at a provider and the
  `SmartModelRouter` (always on) selects the model — you never pass a `model:` field in
  API request bodies.

---

## How configuration is applied

| Deploy target | Config source | Generated by |
|---|---|---|
| Docker Compose | `.env` in the repo root (`docker-compose.yml` reads it) | `install.sh` (`--quick` / wizard / `--env PATH`) or copy `.env.example` → `.env` by hand |
| Kubernetes (Helm) | `helm/openagentic/values.yaml` + your overlay (`-f my-secrets.yaml`) | `install.sh --helm` writes `~/.openagentic/helm-secrets.yaml` |

Start from the template and let the installer fill in secrets:

```bash
cp .env.example .env          # then set the required secrets...
# ...or just let install.sh generate them:
./install.sh --quick          # zero-config Docker path
```

## Compose environment variables

The sections below follow the grouping in `.env.example` and `docker-compose.yml`.

### Secrets (required — no weak defaults)

These are the inter-service trust roots. `docker-compose.yml` requires each via
`${VAR:?...}`, so a bare `docker compose up` aborts immediately if any is unset. The api
also fails **closed** in production on a missing or weak value. Generate each with
`openssl rand -hex 32` (the installer does this for you).

| Variable | Required | Purpose |
|---|---|---|
| `POSTGRES_PASSWORD` | Yes | Postgres superuser password; also embedded in `DATABASE_URL` for api + workflows. |
| `JWT_SECRET` | Yes | Signs/validates user + session JWTs. Must match across api, ui, mcp-proxy, workflows. |
| `SIGNING_SECRET` | Yes | Signs internal HS256 inter-service tokens. Must match across the same services. |
| `INTERNAL_API_KEY` | Yes | api↔mcp-proxy service-account bearer. Also reused as `OPENAGENTIC_PROXY_INTERNAL_KEY` (api→proxy) and as the proxy's `API_INTERNAL_KEY`. Must match across api, ui, workflows, mcp-proxy, proxy. |
| `FRONTEND_SECRET` | Yes | UI session secret (consumed by the `ui` service). |
| `INTERNAL_SERVICE_SECRET` | Yes | Mints/HMAC-verifies the `oa_sys_` inter-service token. The api mints `oa_sys_<HMAC_SHA256(INTERNAL_SERVICE_SECRET, "openagentic-system-token")>`; the mcp-proxy verifies it with the **same** value. Mismatch → system-root MCP calls 401 (fail closed). |
| `ADMIN_SEED_PASSWORD` | Yes (effectively) | First-boot password for the seeded admin user. Compose has a placeholder default of `changeme-on-first-login`; set a real value before first boot. |

> **Why so many must match.** A first install tripped on every one of these. The api
> signs internal JWTs with `JWT_SECRET`/`SIGNING_SECRET`; the mcp-proxy and workflows
> engine validate them. `INTERNAL_API_KEY` is the shared bearer for `/execute-sync`
> (flows), `/call` (MCP tools), and sub-agent dispatch. If they disagree across services,
> tool calls and flow runs 401/503. `install.sh` generates one value per secret and
> writes it once, so everything agrees.

The cloud-secret stub files are unrelated to these trust roots — see
[Cloud MCP credentials](#cloud-mcp-credentials).

### Initial admin user (local auth)

OSS authentication is **local username/password only** — credentials stored in Postgres.
There is no SSO / federated identity. The first admin is seeded on first boot only.

| Variable | Default | Purpose |
|---|---|---|
| `ADMIN_USER_EMAIL` | `admin@openagentic.local` | Seeded admin login email. |
| `ADMIN_SEED_PASSWORD` | `changeme-on-first-login` (placeholder) | Seeded admin password. **Change before first start.** |
| `LOCAL_ADMIN_USERNAME` | `Admin` | Display name for the seeded admin. |
| `ADMIN_REQUIRE_PASSWORD_RESET` | `false` | Whether to force a password change on first login. Left `false` because you already chose the password via `ADMIN_SEED_PASSWORD`. |
| `AUTH_PROVIDER` | `local` | Auth backend. Local only in OSS. |
| `LOCAL_LOGIN_ENABLED` | `true` | Enables the username/password login form (ui service). |
| `MAGIC_BOOT_TOKEN` | (empty) | One-shot autologin token consumed by `install.sh`'s browser-open. Empty in normal startups → `/api/auth/magic` always 401s. |

After boot, log in as the seeded admin:

```bash
curl -sX POST http://localhost:8080/api/auth/local/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin@openagentic.local","password":"<from-wizard>"}'
```

### Ollama (local model runtime)

Ollama is **required** for semantic tool indexing (embeddings). The compose stack ships
its own `ollama` container plus a one-shot `ollama-init` that pulls the embed model on
first boot. Point `OLLAMA_HOST` at a remote Ollama (e.g. a GPU box) to offload chat.

| Variable | Default | Purpose |
|---|---|---|
| `OLLAMA_HOST` | `http://ollama:11434` (compose service) | Ollama endpoint. On macOS the wizard sets `http://host.docker.internal:11434` so containers reach Ollama on the host. Also wired to `OLLAMA_BASE_URL` / `EMBEDDING_OLLAMA_BASE_URL`. |
| `OLLAMA_ENABLED` | `true` | Enables the Ollama provider path in the api. |
| `OLLAMA_EMBED_MODEL` | `nomic-embed-text` | Embedding model (`~270MB`, runs fine CPU-only). Also wired to `OLLAMA_EMBEDDING_MODEL` / `EMBEDDING_MODEL`. |
| `OLLAMA_CHAT_MODEL` | (empty) | Optional chat model to pre-pull into the local compose Ollama so a bare install can chat with no API keys. Empty by default. Ignored when `OLLAMA_HOST` is remote. |
| `EMBEDDING_PROVIDER` | `ollama` | Embedding backend. The wizard's Vertex strategy overrides to `vertex-ai`. |
| `EMBEDDING_DIMENSIONS` | (empty) | Override embedding dimension when a non-default embed model is used. |

The `ollama-init` one-shot pulls `OLLAMA_EMBED_MODEL` always, and `OLLAMA_CHAT_MODEL` only
if set:

```bash
# ollama-init logic (docker-compose.yml)
ollama pull "$OLLAMA_EMBED_MODEL"
[ -n "$OLLAMA_CHAT_MODEL" ] && ollama pull "$OLLAMA_CHAT_MODEL"
```

> **GPU.** The default `ollama` image is CPU-only (portable). For GPU acceleration, layer
> an override: `docker-compose.gpu-nvidia.yml` (Linux + nvidia-container-toolkit) or
> `docker-compose.gpu-wsl.yml` (WSL2). Example:
> `docker compose -f docker-compose.yml -f docker-compose.gpu-wsl.yml up -d`.

### Bootstrap provider (first-boot LLM seed)

On first boot the api seeds **one** LLM provider row when the provider table is empty (the
admin UI owns it thereafter). This is what lets a bare install chat with zero API keys.
Set `BOOTSTRAP_PROVIDER_NAME=` (empty) to opt out.

| Variable | Default | Purpose |
|---|---|---|
| `BOOTSTRAP_PROVIDER_NAME` | `ollama-local` | Seed provider id. Empty disables seeding. |
| `BOOTSTRAP_PROVIDER_DISPLAY_NAME` | `Ollama (local)` | Display name. |
| `BOOTSTRAP_PROVIDER_TYPE` | `ollama` | Provider type (`ollama`, `aws-bedrock`, `vertex-ai`, ...). |
| `BOOTSTRAP_PROVIDER_CONFIG` | `{"endpoint":"<OLLAMA_HOST>"}` | JSON config. The wizard's Bedrock path overrides to `{"region":...}`. |
| `BOOTSTRAP_PROVIDER_DEFAULTS` | `{"chat":"<OLLAMA_CHAT_MODEL>","embedding":"<OLLAMA_EMBED_MODEL>","embeddingDimension":768}` | JSON default model assignments. |
| `SEEDER_VERSION` | (empty) | Registry seeder version. The wizard bumps this (e.g. `6`) on a Bedrock install so the `role='chat'` assignment row is (re)written on boot. |

The chat role is seeded only when a chat model is available (`OLLAMA_CHAT_MODEL` set, or a
Bedrock chat model in `BOOTSTRAP_PROVIDER_DEFAULTS`). When `OLLAMA_CHAT_MODEL` is set
*alongside* a non-Ollama bootstrap provider (Bedrock), the api seeds a **second**,
selectable Ollama chat provider at lower precedence than the bootstrap default.

Example — make AWS Bedrock Claude the default chat (what the wizard's Bedrock strategy writes):

```bash
BOOTSTRAP_PROVIDER_NAME=aws-bedrock
BOOTSTRAP_PROVIDER_DISPLAY_NAME="AWS Bedrock"
BOOTSTRAP_PROVIDER_TYPE=aws-bedrock
BOOTSTRAP_PROVIDER_CONFIG='{"region":"us-east-1"}'
BOOTSTRAP_PROVIDER_DEFAULTS='{"chat":"anthropic.claude-sonnet-4-6","embedding":"nomic-embed-text","embeddingDimension":768}'
SEEDER_VERSION=6
```

### Vector store (pgvector default, optional Milvus) and tool cache

The default vector backend is **pgvector** (Postgres): `isMilvusEnabled()` in `server.ts`
returns `false` unless `MILVUS_HOST` is set or `MILVUS_ENABLED=true`, so a bare
`docker compose up -d` boots healthy with no etcd/minio/milvus and uses pgvector for MCP
tool search and RAG.

```bash
docker compose up -d   # default — pgvector-only, boots healthy as-is
```

Milvus is **optional** — an add-on for high-availability or large-scale RAG. It (with
bundled `etcd` + `minio`) is gated behind the `milvus` profile, and you enable it by
passing the profile **and** flipping the api's Milvus path on:

```bash
MILVUS_ENABLED=true docker compose --profile milvus up -d   # optional Milvus backend
```

When you opt in, the api connects to Milvus on boot and fails loud if it is unreachable —
so only set `MILVUS_ENABLED=true` alongside `--profile milvus` (or a reachable external
`MILVUS_HOST`).

| Variable | Default | Purpose |
|---|---|---|
| `MILVUS_HOST` | (unset → pgvector-only; `milvus` service name when the profile is up) | Milvus host. `isMilvusEnabled()` treats an unset/empty `MILVUS_HOST` as pgvector-only, so setting it (to a reachable Milvus) is the primary switch that turns the Milvus path on. |
| `MILVUS_PORT` | `19530` | Milvus port. |
| `MILVUS_ENABLED` | (unset → pgvector-only by default) | Explicit opt-out: `MILVUS_ENABLED=false` forces pgvector-only even if `MILVUS_HOST` is set. To run Milvus, set `MILVUS_HOST` (e.g. via `--profile milvus`) and leave `SKIP_TOOL_SEMANTIC_CACHE` non-`true`. The Helm chart wires this when `milvus.enabled: true`. |
| `SKIP_TOOL_SEMANTIC_CACHE` | `true` | When `true` the api runs pgvector-only **and** skips tool-embedding generation / RAG indexing. Set it to `false` (and provide `MILVUS_HOST`) to enable the Milvus path. |
| `DISABLE_RAG` | `true` | Disables RAG retrieval on first boot to prevent pipeline stalls from embedding timeouts when Ollama is shared with chat/tools. Re-enable once you have a dedicated embedding endpoint. |

## MCP enablement (compose)

The mcp-proxy spawns the built-in MCP servers as subprocesses. The proxy wires **9**
built-in MCPs: `aws`, `azure`, `gcp`, `kubernetes`, `prometheus`, `loki`, `github`,
`admin`, and `web` (the `web` MCP also handles knowledge storage/web research).

> **`MCPS_ENABLED` is documentary only** — the mcp-proxy does **not** read it as a gate.
> Default: `web,admin,aws,azure,gcp,kubernetes,github,prometheus,loki`. Keep it in sync
> for readability, but flipping it alone does nothing. The **real** per-MCP switch is the
> `OpenAgentic_<NAME>_MCP_DISABLED` flag below, consumed by
> `mcp_manager.initialize_servers`.

All 9 built-ins **spawn** out of the box. `web`/`admin` need no credentials. The cloud
MCPs (`aws`/`azure`/`gcp`) and `github`/`loki` start without credentials too — an
unconfigured one only surfaces a "needs config" / connection error when you actually call
one of its tools.

| Flag | Default | Effect |
|---|---|---|
| `OpenAgentic_WEB_MCP_DISABLED` | `false` | Web research / fetch / knowledge MCP (uses SearXNG). |
| `OpenAgentic_KNOWLEDGE_MCP_DISABLED` | `false` | Knowledge-MCP toggle (knowledge functions live in the web MCP). |
| `OpenAgentic_ADMIN_MCP_DISABLED` | `false` | Platform admin MCP (no creds). |
| `OpenAgentic_AWS_MCP_DISABLED` | `false` | AWS MCP. |
| `OpenAgentic_AZURE_MCP_DISABLED` | `false` | Azure MCP. |
| `OpenAgentic_GCP_MCP_DISABLED` | `false` | GCP MCP. |
| `OpenAgentic_KUBERNETES_MCP_DISABLED` | `false` | Kubernetes MCP (uses mounted `~/.kube`). |
| `OpenAgentic_GITHUB_MCP_DISABLED` | `false` | GitHub MCP (needs `GITHUB_TOKEN` to call tools). |
| `OpenAgentic_PROMETHEUS_MCP_DISABLED` | `false` | Prometheus MCP (wired to in-stack `prometheus:9090`). |
| `OpenAgentic_LOKI_MCP_DISABLED` | `false` | Loki MCP (needs an external `LOKI_URL` to call tools). |
| `OpenAgentic_ALERTMANAGER_MCP_DISABLED` | `true` | Alertmanager MCP (off by default). |
| `SEQUENTIAL_THINKING_MCP_DISABLED` | `true` | Sequential-thinking MCP (off by default). |

### Cloud MCP credentials

The mcp-proxy prefers your **existing host CLI configs**, mounted read-only — the same
auth you already use on your machine:

```yaml
# docker-compose.yml — mcp-proxy volumes
- ${HOME}/.azure:/root/.azure:ro          # az
- ${HOME}/.aws:/root/.aws:ro              # boto3
- ${HOME}/.config/gcloud:/root/.config/gcloud:ro   # google-auth
- ${HOME}/.kube:/root/.kube:ro            # kubernetes
```

It also loads three per-cloud env files (`install.sh` creates empty stubs so the mounts
never fail), letting you paste inline credentials instead of relying on the host CLI:

```yaml
# docker-compose.yml — mcp-proxy env_file
- ${HOME}/.openagentic/cloud-secrets/aws.env
- ${HOME}/.openagentic/cloud-secrets/azure.env
- ${HOME}/.openagentic/cloud-secrets/gcp.env
```

This is what makes "show me my Azure subscriptions" work in 5 minutes — run
`az login` / `aws configure` / `gcloud auth login` on the host, then start the stack.

## Helm values (Kubernetes)

The Kubernetes deployment is configured through `helm/openagentic/values.yaml`. The
default chart runs a self-contained single-replica stack (postgres + redis + ollama + the
app services + searxng), and—unlike compose—defaults to **Milvus on** as the vector
backend. Override `image.*`, `ingress.*`, and `secrets.*` in your own overlay for a real
deployment.

```bash
helm upgrade --install openagentic ./helm/openagentic \
  -n openagentic --create-namespace -f my-secrets.yaml
# ...or let the installer generate + persist secrets:
./install.sh --helm
```

### `ollama`

The in-cluster Ollama **always** serves embeddings (`nomic-embed-text`, CPU-fine). The
chat model is served by `chatHost`: empty uses the in-cluster Ollama (all-local,
zero-config); a `http://HOST:11434` value points chat at a remote/GPU Ollama (the
in-cluster one then pulls only the embed model).

| Key | Default | Purpose |
|---|---|---|
| `ollama.embedModel` | `nomic-embed-text` | Embedding model (in-cluster). |
| `ollama.chatModel` | `llama3.2:3b` | Chat model pulled in-cluster (used when `chatHost` is empty). |
| `ollama.chatHost` | `""` | Remote chat Ollama endpoint. Empty = in-cluster. |
| `ollama.gpu` | `false` | GPU for the in-cluster Ollama. Embeddings never need a GPU; enable only for a big in-cluster chat model. |

`secrets.awsAccessKeyId`/`awsSecretAccessKey` (or a pod IAM role). Embeddings stay on the
in-cluster Ollama.

| Key | Default | Purpose |
|---|---|---|
| `bootstrapProvider.enabled` | `false` | Seed this provider instead of local Ollama. |
| `bootstrapProvider.name` | `aws-bedrock` | Provider id. |
| `bootstrapProvider.displayName` | `AWS Bedrock` | Display name. |
| `bootstrapProvider.type` | `aws-bedrock` | Provider type. |
| `bootstrapProvider.chatModel` | `anthropic.claude-sonnet-4-6` | Default chat model. |

### `securityContext` / `podSecurityContext`

The chart ships a maximally-hardened baseline (CM-7 / AC-3): containers run as non-root
with `allowPrivilegeEscalation: false`, all capabilities dropped, and the
`RuntimeDefault` seccomp profile. Per-workload `overrides` are deep-merged over the
defaults (e.g. `workflows`/`mcpProxy`/`ollama`/`etcd` flip only `runAsNonRoot: false`
where the image has no non-root `USER`; `proxy`/`redis` opt into a read-only root
filesystem). Pod-level overrides pin each image's real uid/gid + `fsGroup` so PVC data
dirs are writable.

| Key | Default | Purpose |
|---|---|---|
| `securityContext.runAsNonRoot` | `true` | Container runs non-root. |
| `securityContext.allowPrivilegeEscalation` | `false` | No privilege escalation. |
| `securityContext.readOnlyRootFilesystem` | `false` | Opt a workload into `true` via its override. |
| `securityContext.capabilities.drop` | `[ALL]` | Drop all Linux capabilities. |
| `securityContext.seccompProfile.type` | `RuntimeDefault` | Seccomp baseline. |
| `securityContext.overrides` | (per-workload) | Deep-merged container-level deltas. |
| `podSecurityContext.runAsNonRoot` | `true` | Pod-level non-root. |
| `podSecurityContext.seccompProfile.type` | `RuntimeDefault` | Pod-level seccomp. |
| `podSecurityContext.overrides` | (per-workload uid/gid/fsGroup) | Pin each image's runtime user + fsGroup. |

Set both `securityContext` and `podSecurityContext` to `null` to disable hardening
entirely (the templates then emit zero securityContext keys).

## Configuration checklist

Before first boot, make sure you have:

- [ ] Set every required secret (or let `install.sh` generate them).
- [ ] Set the same `JWT_SECRET` / `SIGNING_SECRET` / `INTERNAL_API_KEY` /
      `INTERNAL_SERVICE_SECRET` across **all** services (the installer does this).
- [ ] Set `ADMIN_USER_EMAIL` + `ADMIN_SEED_PASSWORD` (don't ship the placeholder).
- [ ] Decided your vector backend: the default `docker compose up -d` is
      pgvector-only and boots healthy as-is — only add `MILVUS_ENABLED=true
      docker compose --profile milvus up -d` if you want the optional Milvus
      backend (Helm defaults Milvus on; set `milvus.enabled: false` for
      pgvector-only there).
- [ ] An Ollama endpoint reachable for embeddings (`nomic-embed-text` will be pulled).
- [ ] At least one chat-capable model — a local Ollama chat model, or an LLM provider key
      / Bedrock bootstrap provider.

Validate after boot:

```bash
docker compose ps                          # all services healthy/running
curl -s http://localhost:8080/api/health | jq .   # api self-reports each dependency
```
